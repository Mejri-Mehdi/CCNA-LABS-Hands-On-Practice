## What I Learned

In this lab, I started with a **pre‑configured OSPF network** (IP addresses and OSPF already set up, but with several issues). My job was to diagnose and fix each problem.

### Issues and fixes

**1. New serial link between R1 and R2**  
   - Configured the interface: `clock rate 128000` (on DCE side) and `no shutdown`.  
   - Added OSPF on both ends: `ip ospf 1 area 0` (or `network` statement).  
   - Verified adjacency formed.

**2. Only R3 has a route to `10.0.2.0/24`**  
   - **Why?** The subnet was likely not advertised into OSPF by the router that owns it (maybe R3 itself, but R3 has it; others don’t).  
   - **Fix:** Ensured the network statement or interface OSPF configuration included that subnet on the router where it resides. Also checked passive interfaces – if the interface is passive, the route is still advertised, so that wasn’t the issue. The problem was likely a missing OSPF enablement on that interface.

**3. R2 and R4 won’t become OSPF neighbors with R5**  
   - **Why?** Possible mismatched OSPF parameters: area ID, hello/dead timers, authentication, network type, or MTU.  
   - **Fix:** Verified on R5 and the adjacent router that both interfaces are in the same area, timers match (`ip ospf hello-interval 10`, `dead-interval 40`), no authentication mismatch, and MTU is consistent. Corrected any discrepancy.

**4. PC1 and PC2 can’t ping external server `8.8.8.8`**  
   - **Why?** No default route to the Internet, or the ASBR not injecting a default route into OSPF, or missing redistribution.  
   - **Fix:** On the router connected to the Internet (likely R1 or R2), added a static default route (`ip route 0.0.0.0 0.0.0.0 <next-hop>`), then redistributed it into OSPF using `default-information originate`. Also checked that the external server is reachable from the ASBR.

**5. Examine the Link State Database (LSDB)**  
   - Used `show ip ospf database` to see LSAs.  
   - **LSAs present included:**  
     - **Type 1 (Router LSA)** – each router’s directly connected links within an area.  
     - **Type 2 (Network LSA)** – generated by the DR on multi‑access networks.  
     - **Type 3 (Summary LSA)** – inter‑area routes (if multiple areas existed).  
     - **Type 4 (ASBR Summary LSA)** – if an ASBR is present.  
     - **Type 5 (External LSA)** – for the default route injected into OSPF.

### Key takeaways

- **Serial links** need a clock rate on the DCE side.
- **OSPF neighbor issues** often stem from mismatched hello/dead timers, area IDs, or MTU.
- **Missing routes** are usually due to omitted OSPF enablement or passive‑interface misconfiguration.
- **Default route injection** is required for Internet access; `default-information originate` (with a static default route) is the standard method.
- **`show ip ospf database`** is a powerful command to understand the full topology and LSA types.

This lab sharpened my systematic troubleshooting skills for OSPF networks.
